## 6. Bugs & Issues Found

### Bug #1 — Kite WebSocket 403 at Market Open (HIGH)
**Symptom:** `WebSocket connection upgrade failed (403 - Forbidden)` from 09:15:00 onward.  
**Root cause:** Zerodha access tokens expire daily and must be regenerated each morning before 09:15. The `ZERODHA_ACCESS_TOKEN` in `.env` was stale from the previous day.  
**Impact:** Missed the entire first 1h 31min of market data. Any signals that would have fired in the morning session were lost.  
**Fix needed:** Add a pre-market token refresh flow (via Kite login URL + redirect) before the 09:15 scheduler trigger. Consider a token-validity check at startup that alerts if the token is expired.

### Bug #2 — `_schedule_tick` errors after market close (MEDIUM)
**Symptom:** From 15:30:00 onward, repeated `RuntimeError: This task group is not active; no new tasks can be started.` errors in `asyncio` callbacks.  
**Root cause:** When the runtime stops at 15:30, `_teardown()` in `KiteIngestor` closes the WebSocket stream, but the Kite library's background thread continues delivering ticks briefly after shutdown. `_on_ws_ticks` uses `loop.call_soon_threadsafe(self._schedule_tick, tick)` without checking if the task group is still alive. By the time the callback runs on the event loop, `self._task_group` is `None`, but the check at [kite_ingestor.py:160](src/trading/engine/kite_ingestor.py#L160) is:
```python
def _schedule_tick(self, raw: Tick) -> None:
    tg = self._task_group
    if tg is not None:
        tg.start_soon(self._handle_tick, raw)  # tg is already torn down
```
The `tg is not None` check passes because `_task_group` is cleared only after the `create_task_group()` context exits, but `tg.start_soon` is called on an already-exited group.  
**Fix needed:** In `_schedule_tick`, guard with a flag (e.g. `self._running`) set to `False` in `_teardown()`, or check `tg.cancel_scope.cancel_called`.

### Issue #3 — SELL signal rejected as ALREADY_IN_POSITION at 14:25 (LOW / investigate)
**Symptom:** SELL signal at 14:25 rejected — but a BUY was only filled at 14:35. If there was no prior open position, this rejection was incorrect.  
**Possible cause:** A position from a prior session (e.g. May 15 or 16) was never squared off in the `positions` table. Paper broker positions persist across restarts.  
**Action:** Query `SELECT * FROM positions WHERE symbol='INFY'` to verify. Add end-of-day position reconciliation or auto-square-off logic.
